# “简阅”小说阅读平台——系统运维操作手册

## 一、 自动化系统巡检与监控自愈 (monitor_yue.sh)
在 openEuler 信创运行环境中，为了保障“简阅”系统 7×24 小时高并发在线运行，必须建立完善的巡检与故障自愈体系。运维人员可以通过运行 `/root/Yue/ops/monitor_yue.sh` 实施周期性巡检。

### 1.1 巡检监控核心模块逻辑
脚本执行后将自动检测以下 5 大核心服务指标：
1. **容器存活检测**：利用 `docker ps --filter "name=yue-app"` 抓取 yue-app 运行时状态，确保系统正在安全封装运行。
2. **资源占用率监控**：通过 `docker stats --no-stream` 拦截容器占用 CPU 算力比、当前独占内存大小、网卡流量等性能参数，提防因 Python 循环或第三方异常导致的溢出。
3. **健康检查与自动容灾自愈（核心亮点）**：
   - 脚本主动对宿主机回环地址调用 `curl -s -f http://127.0.0.1:8000/health`。
   - 若服务正常响应，显示 `后端服务正常响应中 (OK)`。
   - 若发生阻塞或进程崩溃假死，脚本捕获异常后，**自动启动容灾自愈程序**，强制重启受损容器：`docker restart yue-app`。
   - 等待 3 秒恢复连接后，重新检测。若顺利拉起则向管理员信道输出自愈成功提示；若两次检测均告失败，则发出红色最高警报，呼叫人工响应。
4. **日志栈异动追踪**：利用 `docker logs --tail 50 yue-app` 拦截最近 50 行核心输出，快速定位由于底层 SQLAlchemy 发起的数据库错误或死锁。
5. **物理磁盘报警**：通过系统底座的 `df -h` 全面扫描 40GB 系统盘使用配额，预防因容器控制台标准输出暴增，导致文件系统崩溃挂死。

### 1.2 手动巡检方法
运维人员只需登录 openEuler 服务器，直接在控制台调起即可：
```bash
cd /root/Yue/ops
./monitor_yue.sh
```

### 1.3 自动化定时巡检配置 (Cron)
运维人员可将巡检程序注入 openEuler 自带的 crontab 定时调度，设置每 10 分钟自动执行并输出日志归档。执行 `crontab -e` 并写入：
```text
*/10 * * * * /root/Yue/ops/monitor_yue.sh >> /var/log/yue_monitor.log 2>&1
```

## 二、 业务数据容灾备份 (backup_yue.sh)
生产环境下，数据资产安全是运维的生命线。系统通过定时备份机制对业务数据（SQLite 数据库文件及核心工程源码）进行全方位双向归档。

### 2.1 备份流程设计说明
脚本 `/root/Yue/ops/backup_yue.sh` 实现了全自动逻辑：
1. **目录隔离归档**：在宿主机建立安全独立的持久备份根路径 `/root/backups/yue/`。
2. **数据库热拷贝避锁**：由于应用在高并发写入时会持有 SQLite 的独占写锁文件（`.db-journal`），直接在文件系统对 `yue.db` 进行拷贝容易造成文件内容错位损坏。为此，备份脚本利用 Docker 通道 `docker cp` 安全地跨容器导出 `/app/data/yue.db`，重构生成不带事务锁的纯净归档 `yue_YYYYMMDD_HHMMSS.db`。
3. **源码与环境打包**：同步将 `docker-compose.yml`、`Dockerfile`、前端文件 `code_artifact.html`、以及整个 `backend` 逻辑代码压制为一个包含时间戳的压缩包 `yue_project_YYYYMMDD_HHMMSS.tar.gz` 集中归档，确保环境一键可重构。
4. **旧备份淘汰自清理**：由于宿主机仅提供 40GB 系统存储盘，为防止备份无限期暴增引发的物理容量耗尽危机，脚本在备份结束时利用 `find` 查找备份目录下超过 7 天（`+7` 天）的旧历史备份，自动执行物理擦除。

### 2.2 定时自动备份 crontab 排程
根据项目会议纪要决议，每日凌晨 `02:00` 属于系统业务访问的绝对低谷。此时执行数据备份影响最小。在终端运行 `crontab -e` 键入：
```text
0 2 * * * /root/Yue/ops/backup_yue.sh >> /var/log/yue_backup.log 2>&1
```

## 三、 灾后故障一键极速恢复与还原 (restore_yue_db.sh)
当遇到底层虚拟机文件系统硬件故障、突发数据库逻辑损坏，或人为误操作删改数据时，运维人员可在一分钟内进行极速冷备数据库恢复。

### 3.1 恢复指令与参数规范
故障恢复脚本位于 `/root/Yue/ops/restore_yue_db.sh`。执行该命令需要传入**备份数据库文件的完整绝对路径**作为第一参数：
```bash
# 1. 切换到运维路径
cd /root/Yue/ops

# 2. 查看目前本地现存的备份介质列表
ls -lh /root/backups/yue/

# 3. 指定健康的备份文件，一键执行恢复
./restore_yue_db.sh /root/backups/yue/yue_20260604_020000.db
```

### 3.2 恢复过程闭合机制解析
在运维人员发起恢复后，脚本会自动执行以下四步硬核恢复逻辑：
1. **物理强制关停，解脱写锁**：脚本首先执行 `docker-compose down` 强制下线并销毁当前的 `yue-app` 容器，强制回收文件描述符和对数据卷 `yue_data` 的读写锁定。
2. **轻量网格重组，初始化挂载**：调起 `docker-compose up -d` 重新构建、拉起干净的基础虚拟网络与持久挂载卷环境。
3. **数据暴力复制，安全覆盖**：等待容器和内置环境启动 3 秒且数据库进入初始化状态后，调用 `docker cp` 指令，安全且暴力的将健康的备份物理文件覆盖写入容器内部的实盘目标 `/app/data/yue.db`。
4. **连接重载，热启动运行**：执行 `docker restart yue-app` 重启应用。FastAPI 重新加载全新的 SQLite 写句柄。自此数据恢复闭环完成，整个过程客户端访问中断仅约 5-10 秒。
